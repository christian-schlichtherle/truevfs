1. Concurrent Modifications on Windows

Where the issue applies

This section applies to the Windows platform only and exhibits a potential issue with general file I/O. Although this is not an issue with TrueZIP, it is reported here since client applications may experience it when using TrueZIP.

How to reproduce the issue

Compile FileTest.java and run it using the following command in any directory:

java -Djava.io.tmpdir=. FileTest

When started like this, the test program will create a temporary file in the current directory and then in an endless loop overwrite it, delete it, create it as a directory and delete it again. If any of these operations fail, some statistics about all failures so far are printed and the program continues with a newly created temporary file. You have to terminate the program by pressing Ctrl-C to stop it.

If you run the program and do nothing else, everything will work fine and you will see no output. To see some failures, start a bash shell (available as part of the Cygwin environment), change to the same directory and enter the following:

while true; do ls -l > /dev/null; done

This will list the directory in an endless loop. The java application will now occasionally print failure statistics like this:

Minutes elapsed: 5
Failure in run #3069
Total failures to create file: 5
Total failures to delete file: 0
Total failures to create directory: 16
Total failures to delete directory: 79

In this example, 100 failures have been generated within five minutes.

Things to note

  1. This is not a TrueZIP issue: The test program does I/O with the plain java.io package only. TrueZIP isn't used at all!
  2. The likeliness of failures increases with the number of create/delete operations.
  3. The issue seems to be triggered by certain calls within Cygwin. The issue wasn't reproducible with the Windows Explorer (by pressing F5 very fast) or the Windows cmd.exe shell (by using dir in a loop).

How this affects TrueZIP

When processing many archive files, the classes in the package de.schlichtherle.io may throw an IOException or return the boolean value false when a regular file or an archive file is going to be created or deleted. Operations on archive entries are not affected. Unfortunately, failures on archive files may make it impossible to update an archive file when calling File.umount() and loss of some or all data in the archive may be implied.

How to work around the issue

Do not change into a directory and do not list it while some other party is creating/deleting files or directories within it.
